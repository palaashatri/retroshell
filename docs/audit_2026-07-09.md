# RetroShell — Fresh Audit (2026-07-09)

> Supersedes the scoring notes in `implementation_plan.md`. This audit re-verifies the
> "final sprint / 7.05" claims against the **actual code in the working tree today**,
> with `file:line` evidence. Method: direct source reads + `cargo test` on host +
> a Linux Docker build to check the compositor compiles. Nothing here is inferred;
> every VERIFIED/FALSE line is backed by code or a command result run in this session.

## TL;DR

The **applications are genuinely strong** and in several places *better* than the plan credits.
The headline **"we now run our own smithay compositor"** claim is **half-true and misleading**:
real smithay code exists as a *separate* binary, but `retro-shell` itself is architecturally
unchanged — still a single fullscreen winit/wgpu client that paints the whole desktop into one
surface. Two specific feature claims are **false** (PAM lock, theme names). One critical fact is
**unverified in this workspace**: that the compositor actually compiles/runs on Linux.

| "Real smithay compositor" = full Wayland compositor | **PARTIAL** | It is a **nested X11 compositor** (`backend_x11`), i.e. a window inside Xvfb — not a DRM/KMS session compositor. No multi-output, no XWayland, no decorations negotiation. |

| "5 named themes: Classic, Dark, High Contrast, Solarized, Dracula" | **FALSE (names)** | 5 themes exist and persist to `settings.conf` (`theme_manager.rs:449-518`), but they are **Classic, Dark, Grape, Blueberry, Strawberry**. No Solarized / Dracula / High Contrast. |

## ‼️ WAS BROKEN — NOW FIXED (2026-07-09): compositor didn't compile → whole image couldn't build

> **Update:** the one-line fix below was applied to `crates/retro-compositor/src/main.rs:564`
> and verified: `docker build -t retroshell .` → **exit 0**, `retro-compositor` compiles and
> the binary (4.7 MB) is present in the image. The workspace now builds from scratch. Runtime
> verification (compositor actually serving `retro-shell`, not the labwc fallback) is still
> open — see "Still unverified" below.

Originally verified broken via `docker build` (exit code **1**, `cargo build --release
--workspace` exit **101**):

```
error[E0308]: mismatched types
  --> crates/retro-compositor/src/main.rs:564:43
   |
564|   let pos = ev.position_transformed(state.output_size);
   |                ------------------- ^^^^^^^^^^^^^^^^^ expected `Size<i32, Logical>`, found `Size<i32, Physical>`
```

Consequences:
1. **`retro-compositor` has never compiled on Linux.** The "real GL compositor" from the
   "final sprint" commit is broken by a single type error.
2. Because one workspace member fails, **`cargo build --release --workspace` fails entirely**,
   so a **from-scratch `docker build` produces *no* release binaries** — not even `retro-shell`.
   The shipped image cannot be built today.
3. The reason this went unnoticed: `target-docker/` binaries (`retro-shell`, `finder`,
   `terminal`) are dated **Jul 4**, *before* the compositor was added (Jul 6–7). No fresh
   image was ever built after the compositor landed.

**The fix is one line** (see `implementation_plan.md` T0): `position_transformed` wants a
`Size<i32, Logical>`; `output_size` is `Physical`. Construct a logical size from the same
dimensions (scale is 1 under the nested-X11 backend):
```rust
let logical = smithay::utils::Size::<i32, Logical>::from((state.output_size.w, state.output_size.h));
let pos = ev.position_transformed(logical);
```

## Still unverified (blocked on the fix above)

- **Does the compositor actually run and does `retro-shell` connect to it?** No screenshot/

  falls back to labwc if the compositor dies within 3s, so even after the fix a VM run must
  confirm `WAYLAND_DISPLAY` came from `retro-compositor`, not the labwc fallback.
